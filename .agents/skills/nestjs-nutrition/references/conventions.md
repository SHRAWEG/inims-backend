# Naming & Coding Conventions — Single Source of Truth

> Every file name, class name, variable, DB column, and API endpoint must follow these rules. No exceptions.
> All naming rules in this file apply universally — they never change per project. The only thing that changes per project is the domain entity names.

---

## File Naming — All `kebab-case`

| File Type | Pattern | Example |
|---|---|---|
| Entity | `<entity-name>.entity.ts` | `product.entity.ts`, `invoice.entity.ts` |
| Service | `<module-name>.service.ts` | `products.service.ts`, `invoices.service.ts` |
| Controller | `<module-name>.controller.ts` | `products.controller.ts` |
| Module | `<module-name>.module.ts` | `products.module.ts` |
| Create DTO | `create-<entity>.dto.ts` | `create-product.dto.ts` |
| Update DTO | `update-<entity>.dto.ts` | `update-product.dto.ts` |
| Response DTO | `<entity>-response.dto.ts` | `product-response.dto.ts` |
| Query DTO | `query-<entity>.dto.ts` | `query-product.dto.ts` |
| Type | `description.type.ts` | `order-summary.type.ts` |
| Guard | `guard-name.guard.ts` | `jwt-auth.guard.ts` |
| Filter | `filter-name.filter.ts` | `http-exception.filter.ts` |
| Interceptor | `interceptor-name.interceptor.ts` | `response-transform.interceptor.ts` |
| Pipe | `pipe-name.pipe.ts` | `parse-uuid.pipe.ts` |
| Decorator | `decorator-name.decorator.ts` | `current-user.decorator.ts` |
| Strategy | `strategy-name.strategy.ts` | `jwt.strategy.ts` |
| Utility | `description.util.ts` | `date.util.ts` |
| Config | `concern.config.ts` | `database.config.ts` |
| Test | `file-being-tested.spec.ts` | `products.service.spec.ts` |
| Enum | `enum-name.enum.ts` | `order-status.enum.ts` |
| Migration | Auto-generated by CLI | `1710000000000-CreateProductsTable.ts` |

## Class Naming — All `PascalCase`

| Type | Pattern | Example |
|---|---|---|
| Entity | Singular noun | `Product`, `Invoice`, `OrderItem` |
| Service | Plural + Service | `ProductsService`, `OrdersService` |
| Controller | Plural + Controller | `ProductsController` |
| Module | Plural + Module | `ProductsModule` |
| Create DTO | Create + Singular + Dto | `CreateProductDto` |
| Update DTO | Update + Singular + Dto | `UpdateProductDto` |
| Response DTO | Singular + ResponseDto | `ProductResponseDto` |
| Guard | Name + Guard | `JwtAuthGuard` |
| Filter | Name + Filter | `HttpExceptionFilter` |
| Interceptor | Name + Interceptor | `ResponseTransformInterceptor` |
| Pipe | Name + Pipe | `ParseUuidPipe` |
| Strategy | Name + Strategy | `JwtStrategy` |

## Variable / Function Naming — `camelCase`

```typescript
// ✅ Correct
const orderItem = await this.findOneOrFail(id);
const dailyTotal = results.reduce((sum, r) => sum + r.amount, 0);
function buildPaginatedResponse<T>(data: T[], meta: PaginationMeta) { ... }

// ❌ Wrong
const OrderItem = ...;        // PascalCase is for classes
const daily_total = ...;      // snake_case is for DB columns
```

## API Endpoint Naming — REST Rules

### URL Structure

```
# ✅ Correct
GET    /api/v1/products                   # List (plural nouns)
POST   /api/v1/products                   # Create
GET    /api/v1/products/:id               # Get one
PATCH  /api/v1/products/:id               # Update
DELETE /api/v1/products/:id               # Delete

# Nested resources
GET    /api/v1/orders/:id/items
POST   /api/v1/orders/:id/items

# Analytics (noun-based, not verb-based)
GET    /api/v1/analytics/revenue-trend
GET    /api/v1/analytics/category-breakdown

# ❌ Wrong
GET    /api/v1/Product                    # PascalCase in URL
GET    /api/v1/product                    # singular
POST   /api/v1/products/create            # verb in URL
DELETE /api/v1/products/:id/delete         # verb in URL
GET    /api/v1/getProducts                # verb in URL
```

### Rules

| Rule | Correct | Wrong |
|---|---|---|
| Always plural nouns | `/products` | `/product` |
| Always `kebab-case` | `/order-items` | `/orderItems` |
| No verbs in URLs | `DELETE /products/:id` | `/products/:id/delete` |
| Nested resources for children | `/orders/:id/items` | `/items?orderId=:id` |
| Query params in `camelCase` | `?startDate=` | `?start_date=` |

## Database Naming — `snake_case`

### Tables — Plural, snake_case

```sql
CREATE TABLE products (...);
CREATE TABLE order_items (...);
CREATE TABLE audit_logs (...);
CREATE TABLE invoice_line_items (...);
```

### Columns — snake_case

```sql
user_id UUID NOT NULL,
created_at TIMESTAMPTZ NOT NULL,
password_hash VARCHAR(255),
unit_price DECIMAL(10, 2),
is_active BOOLEAN DEFAULT true,
```

### Indexes

```sql
CREATE INDEX idx_products_user_id ON products (user_id);
CREATE INDEX idx_order_items_order_id ON order_items (order_id);
CREATE UNIQUE INDEX idx_users_email ON users (email);
```

### Foreign Keys

```sql
ALTER TABLE order_items
ADD CONSTRAINT fk_order_items_order_id
FOREIGN KEY (order_id) REFERENCES orders (id) ON DELETE CASCADE;
```

### TypeORM Entity ↔ DB Mapping

```typescript
@Entity('order_items')                         // table name: snake_case plural
export class OrderItem extends BaseEntity {     // class name: PascalCase singular
  @Column({ name: 'order_id', type: 'uuid' })  // column name: snake_case
  orderId: string;                               // property name: camelCase

  @Column({ name: 'unit_price', type: 'decimal', precision: 10, scale: 2 })
  unitPrice: number;
}
```

## DTO Rules

### Create DTO

- Contains **all required fields** for creation
- Does NOT include `id`, `createdAt`, `updatedAt`, `deletedAt`
- Validates all fields with `class-validator`

### Update DTO

- Extends `PartialType(CreateDto)` — all fields become optional
- Only fields present in the payload are updated

### Response DTO

- Defines what gets returned to the client
- NEVER returns raw TypeORM entities — always map to response DTO
- NEVER includes sensitive fields (`passwordHash`, `hashedRefreshToken`, etc.)

## TypeScript Rules

| Rule | Correct | Wrong |
|---|---|---|
| No `any` — ever | `data: unknown`, narrow with type guards | `data: any` |
| No non-null assertion `!` without comment | `user!; // safe: checked above` | `user!` |
| Explicit return types on all functions | `async findById(id: string): Promise<User>` | `async findById(id: string)` |
| No magic numbers | `const BCRYPT_ROUNDS = 12;` | `bcrypt.hash(pw, 12)` |
| No magic strings | `enum Status { ACTIVE = 'ACTIVE' }` | `if (status === 'active')` |
| Use `const` by default | `const user = ...` | `let user = ...` (if never reassigned) |
| Prefer `interface` over `type` for objects | `interface UserContext { ... }` | `type UserContext = { ... }` |
| Template literals for interpolation | `` `User ${id} not found` `` | `'User ' + id + ' not found'` |

## Import Order Convention

```typescript
// 1. Node.js built-ins
import * as path from 'path';

// 2. NestJS / framework imports
import { Injectable, Logger } from '@nestjs/common';
import { InjectRepository } from '@nestjs/typeorm';

// 3. Third-party libraries
import { Repository } from 'typeorm';
import * as bcrypt from 'bcrypt';

// 4. Project aliases (@common, @config, @modules)
import { BaseEntity } from '@common/entities/base.entity';
import { AuditLogService } from '@modules/audit/audit-log.service';

// 5. Relative imports (same module)
import { CreateProductDto } from './dto/create-product.dto';
import { Product } from './entities/product.entity';
```

## Adapting Conventions to a New Project

- Domain module names are decided at project start and documented in `references/project-structure.md` or a project-specific README.
- **All naming rules in this file apply universally** — they never change per project, per module, or per entity.
- The only thing that changes per project is:
  - The domain entity names (e.g., `Product` vs `Invoice` vs `Order`)
  - Which optional modules exist (e.g., analytics, notifications, payments)
- When in doubt about naming, apply the patterns from the tables above using your entity name.
